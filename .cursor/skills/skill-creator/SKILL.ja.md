# スキル作成スキル

このスキルは、他のスキルを作成するためのガイダンスを提供します。効果的なスキルの原則、スキルの構造、何を含めるべきか、含めないべきか、そしてスキル作成プロセスについて説明します。

## コア原則

### 簡潔さ

スキルはできるだけ簡潔にする必要があります。長すぎるスキルはコンテキストを圧迫し、パフォーマンスを低下させます。

**ガイドライン:**
- SKILL.mdは通常200-500行以内に収める
- 各リファレンスドキュメントは500行以内
- 不要な繰り返しを避ける
- 例は最小限に、でも効果的に

### 適切な自由度

スキルは正確な指示とClaudeの自己裁量の間でバランスを取る必要があります：

- **高い自由度**: 創造的なタスク（文章、デザイン）
- **低い自由度**: 技術的なタスク（API統合、ファイルフォーマット）

### プログレッシブ開示

情報を段階的に開示して、コンテキストの過負荷を防ぎます：

1. **SKILL.md**: 最も重要な情報のみ
2. **references/**: 必要なときに参照する詳細情報
3. **scripts/**: 実行可能なコード
4. **assets/**: 静的リソース

## スキルの構造

### SKILL.md

すべてのスキルのメインエントリポイント。以下を含みます：

```markdown
---
name: skill-name
description: スキルの簡潔な説明。これがスキルのトリガーに使用されます。
license: LICENSE.txtに完全な条項があります
---

# スキル名

[スキルの概要と目的]

## 使用方法

[基本的な使用手順]

## リファレンス

[詳細なドキュメントへのポインタ]
```

### YAMLフロントマター

```yaml
---
name: skill-name          # スキルの識別子（小文字、ハイフン区切り）
description: |            # スキルのトリガーに使用される説明
  スキルの説明。これはClaudeがいつこのスキルを使用すべきかを
  判断するために使用されます。
license: LICENSE.txtに完全な条項があります
---
```

**重要:** `description`フィールドはスキルのトリガーに非常に重要です。ユーザーのリクエストがこの説明と一致する場合、スキルがアクティブになります。

### scripts/

実行可能なスクリプトを格納します：

- Pythonスクリプト（`.py`）
- Bashスクリプト（`.sh`）
- JavaScriptスクリプト（`.js`）

**命名規則:**
- スネークケース: `convert_to_pdf.py`
- 説明的な名前: `validate_input.sh`

### references/

詳細なドキュメントを格納します：

- 技術仕様
- APIドキュメント
- 例とチュートリアル
- ベストプラクティス

### assets/

静的リソースを格納します：

- テンプレートファイル
- 画像やアイコン
- 設定ファイル
- サンプルデータ

## 何を含めるべきか

### 含めるべきもの

1. **明確な目的**: スキルが何をするかの説明
2. **使用手順**: ステップバイステップのガイド
3. **例**: 最小限だが効果的な例
4. **制約**: 何ができないかの説明
5. **依存関係**: 必要なツールやライブラリ

### 含めないべきもの

1. **一般的な知識**: Claudeがすでに知っていること
2. **冗長な説明**: 同じことを複数回説明しない
3. **過度な例**: 1-2の例で十分
4. **実装詳細**: 必要な場合のみ
5. **個人的な意見**: 客観的な情報のみ

## プログレッシブ開示デザインパターン

### パターン1: 条件付き参照

```markdown
## 基本使用

[基本的な手順]

## 高度な使用

詳細については、`references/advanced-usage.md`を参照してください。
```

### パターン2: 必要に応じて読み込み

```markdown
## トラブルシューティング

問題が発生した場合は、`references/troubleshooting.md`を確認してください。
```

### パターン3: スクリプト実行

```markdown
## セットアップ

セットアップを実行するには：

```bash
bash scripts/setup.sh
```
```

## スキル作成プロセス

### ステップ1: 理解

- ユーザーのニーズを理解する
- 既存のスキルを調査する
- 競合するスキルがないか確認する

### ステップ2: 計画

- スキルの範囲を定義する
- 必要なコンポーネントをリストアップする
- 依存関係を特定する

### ステップ3: 初期化

- ディレクトリ構造を作成する
- SKILL.mdを作成する
- 必要なスクリプトを作成する

### ステップ4: 編集

- SKILL.mdを洗練する
- リファレンスドキュメントを追加する
- スクリプトをテストする

### ステップ5: パッケージ化

- すべてのファイルが正しい場所にあることを確認する
- 不要なファイルを削除する
- READMEを更新する

### ステップ6: 反復

- フィードバックを収集する
- 改善を行う
- バージョンを更新する

## 出力パターン

### テンプレートパターン

厳格または柔軟な要件に対応：

```markdown
## 出力テンプレート

[必須フィールド]
- フィールド1: [説明]
- フィールド2: [説明]

[オプションフィールド]
- フィールド3: [説明]
```

### 例パターン

入力/出力ペアを示す：

```markdown
## 例

**入力:**
```
ユーザー入力の例
```

**出力:**
```
期待される出力の例
```
```

## ワークフロー

### シーケンシャルワークフロー

複雑なタスクをステップに分解：

```markdown
## ワークフロー

1. ステップ1: [説明]
2. ステップ2: [説明]
3. ステップ3: [説明]
```

### 条件付きワークフロー

決定ポイントを通じてガイド：

```markdown
## ワークフロー

1. 条件Aの場合:
   - アクションA1
   - アクションA2
2. 条件Bの場合:
   - アクションB1
   - アクションB2
```

## ベストプラクティス

1. **簡潔に**: 必要最小限の情報のみ
2. **明確に**: 曖昧さを避ける
3. **構造化**: 情報を論理的に整理
4. **テスト**: スキルを実際に使用してテスト
5. **反復**: フィードバックに基づいて改善
