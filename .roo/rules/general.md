## ロール定義

## 期待する回答

- 情報を提示する前に、必ず検証する。明確な証拠なしに推測や憶測をしないこと。
- 変更はファイルごとに行い、間違いを発見する機会を私に与えること。
- 謝罪の言葉は使わないこと。
- 関係のないコードや機能を削除しないこと。既存の構造を維持することに注意してください。
- 変更を提案するときは、コードの性能を考慮し、優先順位をつける。
- コードの変更を修正したり提案したりするときは、常にセキュリティへの影響を考慮する。
- 新しいコードや変更されたコードに対して、適切な単体テストを提案する。
- 必要に応じて、堅牢なエラー処理とロギングを実装する。
- コードの保守性と再利用性を向上させるために、モジュール設計の原則を奨励する。
- 提案された変更が、プロジェクトの指定された言語やフレームワークのバージョンと互換性があることを確認する。
- ロジックを実装するときは、常に潜在的なエッジケースを考慮し、処理する。
- 仮定を検証し、潜在的なエラーを早期に発見するために、可能な限りアサーションを含める。

## 出力に関する注意

- コードのコメントもユーザーからの質問に対する回答も日本語でおこなうこと。
- Mermaid 記法:
  - Markdown ファイル内で Mermaid 図を記述する際は、` ```mermaid ... ``` ` のようにコードブロックで囲むこと。
  - ノードのテキストに括弧 `()` やドット `.` などの特殊文字が含まれる場合は、テキスト全体をダブルクォーテーション `"` で囲むこと。例: `Node["Text with (special) chars."]`
